## Purpose

This document explains how the Go2 jump stack is organized, how data moves through
the system, and which validation steps should be used when debugging or extending
the controller.

The stack is intentionally narrow in scope: it aims to provide a clear simulation
path for low-level forward-jump experiments, not a full locomotion framework.

## Package Responsibilities

- `src/unitree_mujoco`
  Runs the Go2 simulator and publishes the bridge topics required by the controller.
- `src/unitree_ros2`
  Provides the ROS 2 message definitions and support packages used in this
  workspace.
- `src/go2_jump_planner`
  Converts a target distance into a jump plan summary.
- `src/go2_jump_controller`
  Implements the low-level phase machine and publishes `LowCmd`.
- `src/go2_jump_bringup`
  Stores shared parameters and launch descriptions.
- `tools/verify_rt_lowcmd.cpp`
  Validates direct DDS control against the simulator without ROS 2 controller logic.

## Runtime Interfaces

### Simulator Outputs

The MuJoCo bridge publishes:

- `rt/lowstate`
- `/lowstate`
- `/sportmodestate`
- compatibility aliases required by the rest of the workspace

### Controller Inputs

The controller subscribes to:

- `/lowstate`
- `/sportmodestate`
- `/jump_target_distance`

### Controller Outputs

The controller publishes:

- `/lowcmd`

### Trial Artifact

At the end of a run, the controller writes:

- `reports/jump_metrics/latest_report.txt`
- `reports/jump_metrics/latest_trial_context.txt`

The one-shot trial helper also archives timestamped copies in the same directory.
The context file records the active jump profile and the effective launch overrides
used for that trial.

## End-to-End Data Flow

1. `go2_jump_planner` receives the target distance from launch parameters.
2. The planner computes a jump plan and publishes `/jump_target_distance`.
3. `unitree_mujoco` publishes `/lowstate` and bridge state topics.
4. `go2_jump_controller` waits for a stable starting posture.
5. The controller enters the jump phase machine and publishes `/lowcmd`.
6. The simulator reacts to the low-level commands.
7. The controller records metrics during the full motion.
8. A trial report is written when the recovery phase finishes.

## Controller Phase Machine

The controller uses a fixed sequence:

- `stand`
- `crouch`
- `push`
- `flight`
- `landing`
- `recovery`

The controller is parameterized rather than contact-optimized. That makes it easier
to reason about phase behavior and sweep individual parameters.

The latest controller revision keeps the same named phases, but the transitions are
no longer purely time-locked:

- `push` may extend past its nominal duration until takeoff is actually detected
- `flight` may extend past its ballistic estimate until landing is actually detected
- `landing` and `recovery` can use phase-specific damping and optional
  touchdown-hold experiments

## Runtime Configuration Layers

The active controller parameters are resolved in the following order:

1. base YAML defaults in `src/go2_jump_bringup/config/jump_params.yaml`
2. optional named profiles in `scripts/jump_profiles.sh`
3. optional `GO2_JUMP_*` environment-variable overrides
4. final ROS 2 launch arguments generated by `scripts/jump_launch_args.sh`

This layering keeps the repository default stable while still allowing focused
experiments to be reproduced from a single launch command.

## Where the Main Tuning Knobs Act

- `takeoff_speed_scale`
  Adjusts the planned takeoff speed generated from the ballistic distance model.
- `push_front_tau_scale` / `push_rear_tau_scale`
  Redistribute push torque between front and rear legs.
- `push_pitch_target_deg`
  Shapes body pitch during the push phase.
- `flight_pitch_target_deg`
  Shapes body pitch in flight.
- `landing_*`
  Influence landing posture and post-landing recovery behavior.

## Trial Metrics That Matter Most

The report contains many fields, but the following are the most useful during
tuning:

- `final_forward_displacement_m`
  Final settled result.
- `landing_forward_displacement_m`
  Forward position at landing detection.
- `airborne_forward_progress_m`
  True forward progress while airborne.
- `post_landing_forward_gain_m`
  Forward motion added after landing detection.
- `support_hold_forward_gain_m`
  Forward motion accumulated while the controller is still in the landing-support
  hold path.
- `release_to_complete_forward_gain_m`
  Forward motion accumulated after the controller starts releasing from support
  toward the recovery target.
- `max_abs_pitch_deg`
  Useful for spotting overly aggressive pitch shaping.
- `push_extension_after_plan_s`
  Useful for checking whether the nominal push window is still too short.
- `flight_extension_after_plan_s`
  Useful for checking how far the real flight deviates from the ballistic estimate.

As a rule:

- optimize `airborne_forward_progress_m` first
- use `final_forward_displacement_m` as a constraint
- use `post_landing_forward_gain_m` to detect recovery-dominated ŌĆ£fake gainsŌĆØ
- use `support_hold_forward_gain_m` versus `release_to_complete_forward_gain_m`
  to decide whether the next change belongs in touchdown support or late recovery

## Current Known Limitations

- touchdown detection still relies on height and velocity heuristics
- `foot_force_est` is still zero in the present MuJoCo bridge path
- the controller is tuned more strongly for short-hop repeatability than for
  maximum airborne range
- post-landing recovery still contributes a meaningful fraction of the final
  forward displacement
- the current default uses partial touchdown-reference blending, which improves
  landing behavior but still leaves too much nose-down attitude in near-target runs
