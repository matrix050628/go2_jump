# 算法说明

[English](algorithm.md)

这份文档从工程实现的角度说明当前控制栈在做什么。

当前运行主线是：

`JumpTask -> JumpIntent -> WholeBodyMpc -> LowCmd`

这个分层是有意设计的。高层负责回答“跳什么”，低层负责回答“在当前接触状态和当前机体状态下，怎么把这个跳跃意图执行出来”。

## 1. 问题定义

外部任务很简单：

- 输入：目标前跳距离
- 输出：一个干净的前跳，并且大部分前移发生在腾空阶段

这里不允许通过复用 sport mode 内置动作来完成任务。动作必须由我们自己的规划器和低层控制器生成。

这个问题里有两个决定性约束：

- 跳跃的关键时刻高度依赖真实接触状态
- 最终执行层仍然必须落在低层关节命令上

因此，这个项目必须拆成“显式高层规划接口”和“接触感知的低层执行层”两部分。

## 2. `JumpTask` 在表达什么

`JumpTask` 是面向任务层的目标描述。

它包含：

- 目标距离
- 名义起跳角
- 名义起跳速度或起跳速度分量
- 起跳和落地机身俯仰目标
- 名义相位时长

`JumpTask` 故意保持简洁。它表达“目标是什么”，但还没有细化成一份可直接执行的动力学意图。

## 3. `JumpIntent` 在表达什么

`JumpIntent` 是高层规划器输出的显式跳跃意图。

它包含 low-level 能直接执行和约束的量：

- 起跳 `vx` 和 `vz`
- 起跳和落地俯仰目标
- 下蹲、蹬地、腾空、落地、恢复各阶段时长
- 各阶段机身高度偏置
- 收腿系数和落地支撑系数
- 前后腿的足端前后偏置
- 计划顶点高度
- 规划器 backend 名称

`JumpIntent` 是未来 RL planner 的接口边界。

这也是这个仓库最重要的设计决定之一：

高层 RL 输出的应该是一份可解释、可检查、受物理约束的跳跃意图，而不是原始关节力矩，也不是一段黑箱动作片段。

## 4. Planner 在算什么

当前实现：

- 包：`go2_jump_planner`
- backend：`heuristic_explicit`

当前输入：

- `JumpTask`

当前输出：

- 一份显式的 `JumpIntent`

### 4.1 当前 planner 具体在做什么

当前 planner 还是启发式版本，不是学习策略，也不是完整轨迹优化器。

它当前的职责是：

1. 读取名义任务
2. 生成一份自洽的显式跳跃意图
3. 把这份意图发布出来，让 low-level 始终围绕统一接口工作

它现在已经有明确价值：

- 把高层接口固定成显式结构
- 允许我们做 `no_intent` 和 `explicit_intent` 的 A/B 对比
- 给后续 RL planner 留出干净的替换点

### 4.2 后续 RL planner 应该替换什么

未来 RL planner 应替换的是“生成 `JumpIntent` 的策略”，而不是整条控制栈。

也就是说，RL 的输出仍然应该是：

- 起跳速度目标
- 相位时长目标
- 姿态目标
- 收腿和落地准备参数
- 可选的足端偏置参数

换句话说，RL 负责产生“显式 kinodynamic jump intent”，而不是直接输出电机命令流。

## 5. Controller 在算什么

当前实现：

- 包：`go2_jump_mpc`

控制器在每个控制周期都会综合以下信息：

- 当前 `JumpTask`
- 当前 `JumpIntent`
- 当前机器人观测

然后输出下一拍可以执行的低层命令。

### 5.1 观测处理

控制器读取：

- `/lowstate` 的关节位置与速度
- `/lowstate` 的 IMU 姿态和角速度
- `/sportmodestate` 的机身位姿和速度
- 足端相关力信号作为接触代理

接触估计不是一个附属功能，而是整套系统的中心环节。

如果没有接触感知，就无法可靠判断：

- 是否已经真正离地
- 当前是否还在有效蹬地
- 是否已经触地
- 是否可以从落地相位进入稳定恢复

### 5.2 相位管理

控制器不会只相信名义时间轴。

每个控制周期里，它会：

1. 先从任务和意图里采样名义相位参考
2. 再结合当前接触状态和竖直速度
3. 对相位做覆盖修正

这样做是为了避免典型的错拍问题：

- 明明还在支撑，却过早切到 flight
- 还在空中，却过早切到 landing
- 已经触地了，但相位机还没跟上

### 5.3 参考构造

确定当前有效相位后，控制器会构造：

- 目标机身俯仰
- 目标机身高度偏置
- 目标前向速度和竖直速度
- 关节参考位姿
- 分相位的 PD 增益
- 前馈力矩

从代码结构上看，这里就是当前项目里“whole-body control”职能所在的位置。

需要明确的是：

当前仓库还没有一个单独拆开的、标准教科书式的 QP-WBC 模块。现阶段的 low-level 执行包把参考构造、接触相位逻辑和 MPC 搜索放在了一起。

## 6. WBC / MPC 在算什么

当前实现：

- backend 名称：`mujoco_native_mpc`

这个 backend 本质上是一个 MuJoCo rollout MPC。

它还不是完整的 centroidal-QP 求解器，但已经具备 MPC 的核心特征：

在真正发送下一拍低层命令之前，先用同一个 MuJoCo 模型评估短时域后果。

### 6.1 backend 使用的状态

backend 会从当前观测重建 MuJoCo 状态，包括：

- 机身位姿
- 机身速度
- 关节位置
- 关节速度
- 当前接触条件

### 6.2 候选动作

backend 会围绕当前相位参考采样一组结构化候选动作，例如：

- 蹬地伸展强度
- 前向驱动偏置
- 抬升偏置
- 机身俯仰偏置
- 空中收腿幅度
- 落地支撑幅度
- 前后腿推力重分配

然后把每个候选动作放到 MuJoCo 里做短时域 rollout。

### 6.3 代价函数

当前 rollout score 主要考虑：

- 前向速度跟踪
- 竖直速度跟踪
- 俯仰和横滚稳定性
- 接触数与相位的一致性
- 短时域前向推进
- 控制代价

这就是当前项目里的主要低层优化过程。

### 6.4 backend 最终输出什么

backend 对候选动作评估完后，会：

1. 选出当前最优 rollout
2. 只取这个 rollout 的第一拍动作
3. 转成关节参考和前馈量
4. 发布 `LowCmd`

下一个控制周期继续重复这套过程。

## 7. Planner、Controller、WBC/MPC 怎么配合

这三层各自负责不同问题：

- planner：决定一份显式、可解释的跳跃意图
- controller：把这份意图和当前接触状态、机体状态对齐
- WBC/MPC backend：在短时域里选择最合适的低层执行动作

这种分工的意义在于：

- 可以替换 `heuristic_explicit` 为 RL，而不改 `LowCmd` 链路
- 可以继续增强 low-level MPC，而不推翻 `JumpIntent` 接口
- 可以在相同 low-level 栈上做 `no_intent` / `explicit_intent` 对比实验

## 8. 当前主要不足

当前系统已经可运行，也具备研究价值，但还不是最终控制器。

当前最主要的问题仍然是：

- 距离精度不够稳定
- 较大目标距离下的腾空前移仍然不足
- 落地质量和落地后相位干净程度仍然波动较大
- 当前 planner 还是启发式版本
- 当前 low-level backend 还不是完整的接触力优化式 centroidal controller

## 9. 近期主线方向

接下来最重要的一步是：

`RL planner -> explicit JumpIntent -> contact-aware MuJoCo-native MPC -> LowCmd`

这样既能保留系统的可解释性，也能让高层用学习方法替代当前启发式 planner，在跳跃意图选择上获得更强的性能。
